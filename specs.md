# 译界 MVP 需求规格

## 1. 产品目标

“译界”是一款纯客户端的翻译、查词与英文校对工具，以 Web 应用和 Windows 客户端两种形式提供。

首个版本只实现核心语言处理功能，不包含服务端、数据库、登录、用户管理、云同步、计费或管理后台。

最重要的产品原则是：**每次输入都是完全独立的，不得与之前或之后的输入产生任何联系。**

## 2. 交付形态

- Web：响应式静态网页，可部署到任意静态托管平台。
- Windows：使用 Tauri 2 封装，与 Web 端共享 React 前端和业务逻辑。
- 首版优先支持 Windows，代码结构保留未来打包 macOS 的可能性。
- Web 与 Windows 的模型配置互不自动同步。

## 3. 核心语言处理规则

应用默认自动识别当前输入所属的处理模式，并允许用户手动指定模式后重新处理当前输入。

### 3.1 非中英文内容

当输入主体为非中文、非英文语言时：

- 识别输入语言。
- 将内容翻译为自然中文。
- 输出识别到的语言名称和中文译文。

### 3.2 中文单词或短语

当输入为不构成完整句子的中文单词或短语时，输出：

- 最贴合语义的英文词条。
- IPA 音标。
- 词性。
- 恰好 3 个英文例句及对应中文翻译。

### 3.3 中文或中英混合句子

当输入为中文完整句子，或包含中文和英文的混合句子时：

- 输出完整、自然的全英文句子。
- 不得机械保留输入中的英文片段。
- 原有英文内容必须结合整句语境重新组织和表达。
- 仅在存在重要翻译取舍时补充简短说明。

### 3.4 英文单词或短语

当输入为不构成完整句子的英文单词或短语时，输出：

- 规范词形。
- IPA 音标。
- 一个或多个中文释义。
- 词根词缀分析；无法可靠拆分时必须如实说明。
- 同族词及其中文释义。
- 恰好 3 个英文例句及对应中文翻译。

### 3.5 英文句子

当输入为英文完整句子时，检查语法和用词：

- 如果句子正确：明确标记语法和用词正确，并翻译为中文。
- 如果句子有错：输出完整的修正后英文句子，并逐项给出错误片段、替换内容和中文修改原因。

## 4. 输入与结果交互

- 主界面采用左右工作台：左侧输入，右侧展示结构化结果。
- 窄屏设备改为上下布局。
- 支持输入、粘贴、清空、复制结果和重新尝试。
- 支持 `Ctrl/Cmd + Enter` 提交。
- 单次输入最多 5,000 个字符。
- 翻译过程中提供明确加载状态，并允许取消当前请求。
- 显示当前自动识别出的处理模式。
- 用户可在提交前手动选择五种处理模式之一。
- 提供 API 未配置、鉴权失败、额度不足、网络失败、浏览器跨域失败、模型响应异常等错误提示。
- 浏览器跨域失败时，提示用户更换支持 CORS 的接口或改用 Windows 客户端。

## 5. 请求独立性与隐私

- 每次翻译都创建全新的模型请求。
- 请求中只允许包含系统规则和当前输入。
- 不得携带之前的输入、结果、摘要、聊天记录、会话标识或跨请求缓存内容。
- 输出格式失败后的自动重试只能重新发送同一个当前输入。
- 应用不得持久化翻译输入、翻译结果或模型原始响应。
- 刷新页面、关闭应用或清空内容后，当前输入与结果应消失。
- 应用日志不得记录 API Key、输入正文、结果正文或模型原始响应。

## 6. 模型配置

- 用户可以在设置中添加多个模型配置。
- 每个配置包含：显示名称、API Base URL、API Key 和模型标识。
- 首版统一支持 OpenAI-compatible Chat Completions 接口。
- 用户可以在已验证的模型之间切换，并选择默认模型。
- 模型地址、模型标识、显示名称和验证状态可以保存在当前设备。
- 配置内容发生变化后，原验证状态失效，必须重新测试连接。
- 连接测试只校验：接口可访问、鉴权有效、模型存在且返回合法的 Chat Completions 响应。
- 连接测试不得要求模型必须回复某个固定单词或固定文本。
- 只有连接测试成功的配置才可以设为默认模型并用于翻译。

## 7. API Key 存储

### 7.1 Web

- 默认仅将 API Key 保存在当前标签页的 `sessionStorage` 中。
- 用户可主动选择“在此浏览器记住密钥”。
- 开启记忆后，API Key 保存在当前浏览器的 `localStorage` 中。
- 开启前必须明确提示：密钥将以明文形式保存在浏览器本地存储，只建议在私人设备使用。
- Web 端只能直接调用允许浏览器跨域请求的模型接口。

### 7.2 Windows

- API Key 必须保存在 Windows 凭据管理器中。
- API Key 不得写入普通配置文件、源码或浏览器存储。
- Windows 客户端使用 Tauri 原生 HTTP 请求访问模型接口，以避免浏览器 CORS 限制。

## 8. 模型请求与结构化输出

- 模型调用温度应保持较低，以提高翻译和结构化输出稳定性。
- 模型必须只返回 JSON，不得包含 Markdown 代码围栏或额外说明。
- 客户端应兼容并清理模型偶尔返回的 JSON 代码围栏。
- 五种模式分别使用明确的 TypeScript 数据类型和运行时结构校验。
- 自动模式由模型根据当前输入判断任务类型，并返回 `detected_mode`。
- 手动模式下，模型不得擅自切换为其他模式。
- JSON 无法解析或结构不完整时，针对当前输入自动重试一次。
- 第二次仍失败时，停止请求并显示可重新尝试的格式错误。

## 9. 非目标范围

首个版本明确不实现：

- 服务端代理或服务端模型密钥。
- 用户注册、登录和用户管理。
- 云端翻译历史或本地翻译历史。
- 跨设备同步。
- 用量计费、套餐和额度管理。
- 管理后台。
- 多人协作或分享。
- 离线翻译模型。

## 10. 验收标准

- Web 和 Windows 客户端均能使用用户配置的兼容模型完成五类语言处理任务。
- 中文词条和英文词条模式均稳定返回恰好 3 个例句。
- 中英混合句能够生成完整自然英文，不机械保留原有英文片段。
- 正确英文句子返回中文翻译；错误英文句子返回修正与逐项说明。
- 自动识别和手动模式均能返回对应的结构化结果。
- 连续处理两个不同输入时，第二个网络请求中不存在第一个输入的任何内容。
- 输入与结果不出现在 `localStorage`、`sessionStorage`、普通应用配置或日志中。
- Web 能明确处理并提示 CORS 错误。
- Windows API Key 通过系统凭据管理器保存。
- 模型返回错误 JSON 时会自动重试一次，且不会引入其他输入内容。
- Web 生产构建、单元测试和类型检查全部通过。
